# **2단계: 문제 및 가이드 제시**

### 🚀 문제 15: 세 수의 합 (3Sum)

### 🔗 **링크:** [https://leetcode.com/problems/3sum/](https://leetcode.com/problems/3sum/)

## **📜 문제 설명 (Mission Briefing):**

* **입력(Input):** 정수 배열 `nums`가 주어져. (예: `[-1, 0, 1, 2, -1, -4]`)
* **출력(Output):** 배열 안에서 숫자 3개를 골라서 합이 **`0`**이 되는 모든 조합(Triplet)을 리스트로 묶어서 리턴해.
* **🔥 핵심 규칙:** 인덱스는 다 달라야 하고, **중복된 조합이 결과에 들어가면 절대 안 돼.** (예: `[-1, 0, 1]`을 찾았는데 뒤에서 또 `[0, 1, -1]`을 넣으면 오답 처리됨)

## **🎯 오늘 부술 for문 (Big-O 견적):**

* **입력 크기 역산:** `nums`의 길이 $N$이 최대 3000이야.
* **무지성 풀이 ($O(N^3)$):** 아무 생각 없이 "숫자 3개 뽑는 거네?" 하고 3중 for문(`i`, `j`, `k`)을 돌리면? 연산량이 $3000^3 = 2.7 \times 10^{10}$(270억)번이 돼. 파이썬에서는 10초 넘게 걸려서 타임아웃으로 장렬하게 전사해.
* **해결책 ($O(N^2)$):** 그래서 오늘은 **1. 배열을 먼저 정렬($O(N \log N)$)**하고, **2. for문 하나로 첫 번째 숫자를 고정($O(N)$)**한 뒤, **3. 나머지 두 숫자를 투 포인터($O(N)$)**로 찾을 거야. 결론적으로 $O(N^2)$으로 타협 봐서 1초 컷으로 통과시킬 거다.

## **🛠️ 네가 해야 할 것 (논리 설계 힌트):**

형님, 통계에서 다중회귀 돌릴 때 변수 3개($X, Y, Z$)가 얽혀있으면 해석하기 빡세지? 그럴 때 어떻게 해? **변수 하나($X$)를 통제(Control)해서 고정해버리고**, 나머지 두 변수($Y, Z$)의 관계만 파고들잖아.

알고리즘도 똑같아. $X + Y + Z = 0$ 이라는 방정식을 풀어야 하는데, 변수가 3개라 $O(N^3)$이 걸리는 거야.
그럼 **for문 하나로 $X$를 딱 고정**해버리면 어떻게 될까?
우리가 찾아야 할 목표는 **$Y + Z = -X$** 가 되지?

어? 숫자 두 개 더해서 특정 타겟($-X$) 만드는 거... 이거 어디서 많이 보지 않았어?
맞아, 엊그제 푼 **'두 수의 합'**이랑 어제 배운 **'투 포인터'**를 섞어 쓰는 거야.

**[형님이 뚫어야 할 미션]**

1. **투 포인터의 전제 조건:** 투 포인터가 양쪽에서 좁혀 들어오려면 배열이 어떤 상태여야 할까? (이거 먼저 처리해야 해)
2. **$O(N^2)$ 설계:** $X$를 고정하는 바깥쪽 for문 하나, 그리고 그 안에서 $Y, Z$를 찾는 투 포인터 while문 하나.
3. **🔥 헬게이트 오픈 (중복 제거):** 배열 안에 `-1`이 여러 개면 `[-1, 0, 1]` 이라는 정답이 중복으로 여러 번 튀어나올 수 있어. 어떻게 하면 이 중복된 숫자들을 **탐색 과정에서 스킵**할 수 있을지 치열하게 고민해 봐.


5. **🔥 중복 제거 (제일 많이 틀리는 구간):** * 정답을 찾았다고 끝이 아니야. `left` 오른쪽이나 `right` 왼쪽에 똑같은 숫자가 연속으로 있을 수 있어. `while`문 써서 똑같은 숫자는 다 건너뛰게 만들어 주고, 그다음 `left`랑 `right`를 한 칸씩 더 좁혀줘야 해.

### 1. 정렬은 왜 시키는 건데? (Why Sort?)

형님, 시계열 데이터 분석할 때 시간순으로 정렬(Sort) 안 하고 랜덤하게 섞인 상태로 자기회귀(AR) 모델 돌려? 절대 안 그러지. 데이터에 **'방향성(Trend)'**이 있어야 모델이 학습을 하잖아.

알고리즘도 똑같아. 투 포인터라는 기술은 **"데이터가 정렬되어 있다는 믿음"** 하에서만 작동하는 마법이야.

* **방향성 부여:** 오름차순으로 정렬(`[-4, -1, -1, 0, 1, 2]`)을 딱 해두면 뇌에 룰이 하나 생겨. **"왼쪽으로 갈수록 작아지고, 오른쪽으로 갈수록 커진다."** 이 절대 법칙이 있어야 이따가 투 포인터를 움직일 때 "값이 모자라네? 포인터를 오른쪽으로 넘겨서 키워야지!" 하고 방향을 결정할 수 있어. 섞여 있으면 포인터를 어디로 옮길지 완전 랜덤 워크(Random Walk)가 돼버리잖아.
* **중복 제거 (🔥핵심):** 이 문제 조건 중에 "중복된 조합은 넣지 마라"가 있었지? 정렬을 해버리면 똑같은 숫자들(`-1`, `-1`)이 옆으로 찰싹 달라붙게 돼. 그럼 나중에 for문이나 while문 돌릴 때 "어? 방금 본 숫자랑 똑같네? 스킵(`continue`)!" 하고 넘어가기가 엄청 쉬워져.

---

### 2. 그럼 투 포인터는 왜 필요한데? (Why Two Pointers?)

이게 바로 오늘 우리가 부숴야 할 **"for문 파괴"**의 핵심이야.

우리가 풀어야 할 방정식은 $X + Y + Z = 0$ 이야.
이걸 무지성 3중 for문으로 풀면,

1. $X$ 고르고 (for문)
2. $Y$ 고르고 (for문)
3. $Z$ 고르고 (for문) $\rightarrow$ 이러면 시간복잡도가 $O(N^3)$이 돼서 270억 번 연산하다 서버가 터져.

그래서 우리는 아까 말했듯 **$X$ 하나를 통제 변수(고정)로 잡아버릴 거야.** 첫 번째 for문으로 $X$를 딱 하나 박아두면, 이제 문제는 **$Y + Z = -X$** 라는 두 수의 합 찾기 문제로 다운그레이드 돼.

**자, 여기서 투 포인터가 등판해!**
만약 $Y, Z$를 찾을 때 또 이중 for문을 쓰면 여전히 전체는 $O(N^3)$이야. 이걸 찢어버리려고 정렬된 배열 양 끝에 `left`($Y$ 후보)와 `right`($Z$ 후보)를 세우는 거지.

* **`left`**는 제일 작은 값, **`right`**는 제일 큰 값이야.
* 세 수를 다 더해봤어 ($X + Y + Z$)
* **만약 합이 0보다 작다?** $\rightarrow$ 값을 키워야 하잖아? 정렬되어 있으니까 작은 값을 가리키는 `left`를 오른쪽으로 한 칸 옮기면 값이 커지겠지!
* **만약 합이 0보다 크다?** $\rightarrow$ 값을 줄여야 하잖아? 큰 값을 가리키는 `right`를 왼쪽으로 한 칸 옮기면 값이 작아지겠지!

이렇게 양쪽에서 조여 들어오면, $Y$와 $Z$를 찾기 위해 남은 배열을 다 뒤질 필요 없이 **딱 한 번만 훑고 지나가면 끝나버려 ($O(N)$).** 결과적으로 $X$를 고정하는 바깥 for문 $O(N)$ $\times$ $Y, Z$를 찾는 투 포인터 while문 $O(N)$ = **최종 시간복잡도 $O(N^2)$** 으로 타협을 보는 거지! $O(N^3)$ 짜리를 $O(N^2)$으로 확 줄여주는 일등 공신이 바로 이 투 포인터야.

---

어때, 왜 정렬부터 때리고 양쪽 끝에 포인터를 세우는지 수리적 견적이 좀 나와?
이 '방향성을 이용한 탐색' 원리를 머릿속에 딱 넣고 아까 그 스켈레톤 코드의 `while`문 안쪽(`left += 1`, `right -= 1`)을 다시 쳐다봐. 느낌이 확 다를 거다! 가보자고!

### 🚨 팩폭 1: 합이 0일 때(정답 찾았을 때) 중복 처리가 빠졌다!

형님 코드를 보면 합이 0일 때 `results`에 추가하고 `Y_idx += 1`, `Z_idx -= 1`을 딱 한 칸씩만 움직였지?
이러면 어떤 대참사가 벌어지냐면, 배열이 `[-2, 0, 0, 2, 2]` 일 때를 상상해 봐.

1. `X = -2`, `Y = 0(첫번째 0)`, `Z = 2(마지막 2)` $\rightarrow$ 합이 0! 정답 `[-2, 0, 2]` 저장.
2. 형님 코드대로 `Y` 한 칸 올리고, `Z` 한 칸 내림.
3. 다음 턴: `X = -2` (그대로), `Y = 0(두번째 0)`, `Z = 2(앞쪽 2)` $\rightarrow$ 어? 또 합이 0이네? 정답 `[-2, 0, 2]` **또 저장.**

결과에 `[[-2, 0, 2], [-2, 0, 2]]` 이렇게 똑같은 조합이 두 번 들어가서 리트코드에서 **"Wrong Answer (중복 발생)"**로 바로 빠꾸먹어.
**해결책:** 정답을 찾았을 때도, 다음 숫자가 지금 숫자랑 똑같으면 다 스킵해버려야 해!

---

### 🚨 팩폭 2: R과 파이썬의 차이 (IndexError 지뢰)

형님이 `sum > 0` 이나 `sum < 0` 일 때 중복을 건너뛰려고 짠 이 while문 봐봐.

```python
while(nums[Z_idx] == nums[Z_idx+1]):
    Z_idx -= 1

```

이거 논리는 아주 훌륭한데, **파이썬에서는 시한폭탄이야.** R에서는 인덱스 넘어가면 그냥 `NA` 뱉고 말지만, 파이썬은 배열 범위 벗어나면 `IndexError: list index out of bounds` 띄우면서 서버가 터져버려.
만약 `Z_idx`가 계속 빼기(`-1`) 되다가 `0`을 뚫고 지하로 내려가거나 `Y_idx`를 침범하면 어떻게 될까? 그래서 while문 안에서 인덱스를 움직일 때는 **무조건 `Y_idx < Z_idx` 같은 경계 조건(Boundary Condition)**을 같이 걸어줘야 해.

*💡 꿀팁:* 사실 `sum > 0` 이나 `sum < 0` 일 때는 굳이 while문으로 중복 스킵 안 하고 딱 한 칸만 움직여도 돼. 어차피 밖의 메인 `while`문이 다시 계산해서 크면 또 줄여줄 거거든! **중복 스킵은 정답(sum == 0)을 찾았을 때만 빡세게 해주면 돼.**

---

### 🛠️ PT쌤의 교정 타임 (Pythonic + 안정성)

형님 로직을 100% 살리면서, 방금 말한 에러 안 터지게 딱 다듬은 버전이야. 눈으로 슥 봐봐.

```python
from typing import List

class Solution:
    def threeSum(self, nums: List[int]) -> List[List[int]]:
        results = []
        nums.sort() # 1. 정렬 필수
        
        for X_idx in range(len(nums) - 2):
            # X 중복 스킵 (형님이 완벽하게 짠 부분!)
            if X_idx > 0 and nums[X_idx-1] == nums[X_idx]:
                continue
            
            Y_idx = X_idx + 1
            Z_idx = len(nums) - 1
            
            while Y_idx < Z_idx:
                total = nums[X_idx] + nums[Y_idx] + nums[Z_idx]
                
                if total < 0:
                    Y_idx += 1 # 작으면 키운다 (한 칸만)
                elif total > 0:
                    Z_idx -= 1 # 크면 줄인다 (한 칸만)
                else:
                    # 빙고! 정답 추가
                    results.append([nums[X_idx], nums[Y_idx], nums[Z_idx]])
                    
                    # 🔥 [팩폭 1 해결] 양쪽 포인터 중복 스킵
                    # Y가 다음 놈이랑 똑같으면 계속 스킵 (IndexError 방지 조건 추가)
                    while Y_idx < Z_idx and nums[Y_idx] == nums[Y_idx + 1]:
                        Y_idx += 1
                    # Z가 이전 놈이랑 똑같으면 계속 스킵
                    while Y_idx < Z_idx and nums[Z_idx] == nums[Z_idx - 1]:
                        Z_idx -= 1
                        
                    # 중복 다 피했으니 마지막으로 한 칸씩 이동!
                    Y_idx += 1
                    Z_idx -= 1
                    
        return results

```

어때? 형님이 짠 뼈대에서 크게 안 벗어났지?
$O(N^3)$을 $O(N^2)$으로 부순다는 그 **'알고리즘적 사고(Big-O 견적)'**를 스스로 해냈다는 게 오늘 훈련의 찐 성과야. 이 감각 잊지 마.
