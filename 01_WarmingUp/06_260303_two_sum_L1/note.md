# **2단계: 문제 및 가이드 제시**

### 🚀 문제 1: 두 수의 합 (Two Sum)

### 🔗 **링크:** [https://leetcode.com/problems/two-sum/](https://leetcode.com/problems/two-sum/)

## **📜 문제 설명 (Mission Briefing):**

* **입력(Input):** 정수 배열 `nums`와 타겟 정수 `target`이 주어져.
* **출력(Output):** 배열 안에서 두 숫자를 더해 `target`이 되는 두 숫자의 '인덱스'를 리스트로 묶어서 리턴하면 돼.
* **규칙:** 정답은 무조건 딱 하나만 존재하고, 같은 요소를 두 번 쓸 수는 없어.

## **🎯 오늘 부술 for문 (Big-O 견적):**

* **입력 크기 역산:** 문제에서 `nums`의 길이 $N$이 최대 $10^4$야. 아무 생각 없이 $i$랑 $j$ 뽑는 이중 for문 돌리면 $O(N^2)$이니까 최악의 경우 $10^8$(1억)번 연산해. 파이썬에서는 1초 언저리 아슬아슬하게 통과하거나 서버 상태 안 좋으면 타임아웃 터지는 찝찝한 수치지.
* **해결책:** 그래서 오늘은 이 이중 for문을 **딕셔너리(Hash Map)**를 써서 단일 for문 한 번, 즉 시간 복잡도를 $O(N)$으로 확 찢어버릴 거야.

## **🛠️ 네가 해야 할 것 (Step-by-Step 설계):**

코딩하기 전에 논리부터 챙겨. 이 문제는 "내가 필요한 짝꿍이 이미 지나왔던 길에 있었나?"를 확인하는 게 핵심이야.

1. **기록장(Dictionary) 준비:** 숫자의 '값'을 Key로, '인덱스'를 Value로 저장할 빈 딕셔너리를 하나 만들어 둬.
2. **순회 (단일 for문):** 리스트를 돌면서 현재 숫자(`num`)와 현재 인덱스(`i`)를 확인해.
3. **타겟 방정식 풀기:** 우리가 찾아야 할 짝꿍 숫자는 `target - num`이야. (방정식 $X + Y = Target \rightarrow Y = Target - X$ 푸는 거랑 똑같지?)
4. **기록장 조회:** 이 짝꿍 숫자(`Y`)가 우리가 만들어둔 '기록장'에 있는지 조회(Look-up)해.
* **있으면 (Hit):** 빙고! 기록장에 적힌 인덱스와 내 현재 인덱스 `i`를 묶어서 바로 리턴해.
* **없으면 (Miss):** 나중에 뒤에 나올 다른 숫자가 나를 짝꿍으로 찾을 수 있으니, 내 값(`num`)을 Key로, 내 위치(`i`)를 Value로 기록장에 저장해.



## **🧰 필요한 파이썬 내장 함수/문법 (범용 사용법):**

파이썬 초보 티 벗으려면 `enumerate`랑 딕셔너리 `in` 탐색은 숨 쉬듯 써야 해.

* **`enumerate()`**: 리스트 돌 때, 값만 빼오는 게 아니라 **인덱스**도 같이 뽑아주는 꿀함수야.
```python
# 범용 예시
sample_list = ['A', 'B', 'C']
for idx, val in enumerate(sample_list):
    print(f"인덱스 {idx}에 {val}이 있습니다.")

```


* **딕셔너리(Dictionary) 검색**: `in` 연산자로 딕셔너리의 'Key'가 존재하는지 찾을 때, 해시 테이블 구조 덕분에 시간 복잡도는 $O(1)$이야. (리스트에서 `in` 쓰면 처음부터 다 뒤져야 해서 $O(N)$이 걸리니까 주의!)
```python
# 범용 예시
my_dict = {10: "apple", 20: "banana"}
if 10 in my_dict:
    print("10이라는 키가 존재함!") # O(1)에 바로 찾아냄

```

### 🚨 1. 미리 딕셔너리를 꽉 채우면 안 되는 이유 (논리적 함정)

주석 보니까 처음에 `seen`을 한 방에 다 채워놓고 시작하려고 했지?
파이썬 문법으로는 **딕셔너리 컴프리헨션(Dictionary Comprehension)**이라고 해서 `{num: i for i, num in enumerate(nums)}` 이렇게 쓰면 한 줄로 쫙 만들 수는 있어.

**근데 이 문제에선 그렇게 하면 대참사가 나.** 왜냐? 통계로 치면 변수 매핑할 때 **'중복 데이터 덮어쓰기'**랑 **'자기 자신 참조'** 오류가 터지거든.

Test Case 3번 `nums = [3, 3]`, `target = 6`을 생각해 봐.
만약 딕셔너리를 미리 다 만들면, 첫 번째 `3`(인덱스 0)을 두 번째 `3`(인덱스 1)이 덮어써서 `seen = {3: 1}` 딱 하나만 남아버려. 게다가 현재 숫자가 `3`일 때 짝꿍 `3`을 찾으면, 그게 내 자신인지 진짜 다른 짝꿍인지 구분이 안 가잖아?

그래서 형님이 짠 것처럼 **"반복문을 돌면서 과거의 발자취(seen)만 뒤져보고, 못 찾았을 때만 내 기록을 추가하는 방식"**이 이 문제의 유일한 해법이야. 시계열 분석할 때 자기회귀(AR) 모델이 과거(Lag) 데이터만 참조하는 거랑 똑같은 논리라고 보면 돼.

### 💅 2. Pythonic & Efficiency (코드 다듬기)

로직은 완벽한데, 코드를 조금 더 날렵하고 파이썬답게 깎아줄게.

* **불필요한 `break`와 `else` 제거:** `return`을 만나면 함수는 그 즉시 종료돼. 그러니까 `return` 밑에 있는 `break`는 영원히 실행 안 되는 데드 코드(Dead Code)야. 그리고 `if`에서 `return`으로 빠져나가기 때문에 굳이 `else`로 묶어줄 필요 없이 들여쓰기를 한 칸 당길 수 있어.
* **`sorted()` 제거 (엔지니어링 디테일):** 형님이 정답으로 `sorted([i, seen[pair_num]])`를 리턴했는데, 굳이 정렬 함수를 돌릴 필요가 없어. 왜냐? `seen[pair_num]`은 내가 과거에 지나온 인덱스고, `i`는 현재 내 인덱스잖아. 무조건 `seen[pair_num]`이 `i`보다 작을 수밖에 없지! 그냥 `[seen[pair_num], i]` 순서로 리턴하면 정렬 비용을 아낄 수 있어.

**[✨ PT쌤의 깔끔한 모범 답안]**

```python
class Solution:
    def twoSum(self, nums: List[int], target: int) -> List[int]:
        seen = {}
        for i, num in enumerate(nums):
            pair_num = target - num
            
            if pair_num in seen: # O(1) 룩업!
                return [seen[pair_num], i] # 과거 인덱스, 현재 인덱스 순서
            
            seen[num] = i # 못 찾으면 내 기록 남기기

```
