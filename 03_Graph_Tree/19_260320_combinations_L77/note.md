# 2단계: 문제 및 가이드 제시 (The 'For-loop' Perspective)

### 🚀 문제 77: Combinations
### 🔗 **링크:** [https://leetcode.com/problems/combinations/](https://leetcode.com/problems/combinations/)


## **📜 문제 설명 (Mission Briefing):**
두 정수 $n$과 $k$가 주어져. $1$부터 $n$까지의 숫자 중에서 **$k$개의 숫자를 뽑는 모든 가능한 조합(Combinations)**을 리스트로 담아 리턴하는 문제야. 순서가 다르면 같은 취급이야! (통계학의 $nCr$ 비복원 추출 알지?)

* **예시 1:** `n = 4, k = 2` (1, 2, 3, 4 중에서 2개 뽑기)
  * 출력: `[[1,2], [1,3], [1,4], [2,3], [2,4], [3,4]]` (총 $4C2 = 6$개)
* **예시 2:** `n = 1, k = 1`
  * 출력: `[[1]]`

## **🎯 오늘 부술 for문 (Big-O 관점):**
문제 제약 조건을 보면 $n$은 최대 20, $k$는 최대 20이야. $k$가 유동적이니까 무지성 $k$중 for문은 애초에 불가능하지. 어제 배운 재귀(DFS)로 $k$ 깊이만큼 트리를 파고들어야 해. 

근데 여기서 진짜 중요한 최적화 포인트! 순열($nPr$)은 `[1, 2]`랑 `[2, 1]`이 다르니까 앞뒤로 다 탐색해야 했고, 그래서 `visited` 장부(공간 복잡도 $O(N)$)가 필수였어. 하지만 조합($nCr$)은? **"무조건 내(현재 뽑은 숫자) 뒤에 있는 숫자들만 본다"**는 규칙을 걸면 돼! 1을 뽑았으면 다음엔 2, 3, 4만 보고, 2를 뽑았으면 3, 4만 보는 거지. 
이 단순한 아이디어 하나로 불필요한 탐색을 가지치기(Pruning)하고, `visited` 배열 없이 공간 복잡도를 $O(k)$(콜스택 깊이)로, 시간 복잡도를 $O(k \cdot \binom{n}{k})$로 기적처럼 찢어버릴 수 있어. 

## **🛠️ 네가 해야 할 것:**
1. **상태(State) 정의하기:** 어제 순열은 `dfs(path)`만 넘겼지만, 오늘은 "이번 반복문은 몇 번 숫자부터 시작할지" 알려주는 `start` 변수를 추가해야 해. 즉, `dfs(start, path)`가 돼야 함.
2. **종료 조건(Base Case):** 수첩(`path`)에 $k$개의 숫자가 다 적혔다면? 정답 배열에 복사(`[:]`)해서 넣고 빽(`return`)!
3. **가변 for문 (백트래킹):** `start`부터 $n$까지 순회하면서, `path`에 숫자를 넣고, 재귀 함수를 부를 땐 다음 시작점을 `i + 1`로 넘겨준 뒤, 다시 `pop()`으로 빼는 리듬을 타라.

## **🧰 필요한 파이썬 내장 함수/문법(범용사용법):**
**`range()` 함수의 범용 사용 (끝자리 주의)**
파이썬의 `range(시작, 끝)`은 항상 '끝' 숫자의 **직전**까지만 돌아. $1$부터 $n$까지 포함해서 반복문을 돌리려면 무조건 `n + 1`을 써야 해. 코테 볼 때 이거 헷갈려서 인덱스 에러 나는 사람 오조오억명이다.

```python
n = 4
# 1부터 4까지 순회하고 싶을 때
for i in range(1, n + 1):
    print(i) # 1, 2, 3, 4 출력됨
```

### 🧠 네 코드가 찢어버린 것들
1. **`dfs(i + 1, path)`의 마법:** 어제는 `dfs(path)`만 넘기면서 매번 처음부터 다시 뒤졌지? 오늘은 `i + 1`을 넘기면서 **"내가 방금 뽑은 공(i) 다음 번호부터만 볼게!"**라고 선언했어. 이 한 줄 덕분에 무지성 탐색이 싹 날아가고, $O(N)$의 공간을 차지하던 `visited` 배열도 완벽하게 소거됐어.
2. **`path.pop()` 위치:** 재귀를 타고 바닥까지 찍고 온 다음, 방금 책상에 올렸던 공을 치우는 위치가 정확히 for문 안쪽 `dfs()` 바로 밑에 위치했어. 공간 복잡도를 $O(k)$ (재귀 콜스택 깊이)로 묶어둔 1등 공신이야.

> 🗣️ 면접관: "코드 잘 봤습니다. 근데 만약 $n=100, k=99$인데, `start`가 50부터 시작하는 상황이면 어차피 99개를 절대 못 채우잖아요? **for문이 헛도는 걸 막기 위한 최적화(가지치기, Pruning)**를 하나 더 추가해 볼 수 있을까요?"

이때 통계학도의 짬바를 살려서 for문 범위를 이렇게 깎아버리면 면접관들 기립 박수 친다.

```python
# 현재 path에 담긴 개수를 빼면, 앞으로 '더 뽑아야 할 개수'가 나옴
need = k - len(path)

# n에서 need만큼 뺀 숫자까지만 탐색해도 충분함! (그 뒤부터는 어차피 개수 부족)
# range(start, n + 1)  -> 이렇게 썼던 걸 아래처럼 바꿈
for i in range(start, n - need + 2):
```
이걸 적용하면 불필요한 재귀 트리 가지가 댕강댕강 잘려 나가면서 실행 속도가 비약적으로 펌핑돼. (지금 당장 외울 필요는 없고, "아, 표본 크기를 못 채우는 구간은 아예 탐색을 안 할 수도 있겠구나" 정도로만 이해해 둬!)
