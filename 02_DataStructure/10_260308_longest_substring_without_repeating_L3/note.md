# **2단계: 문제 및 가이드 제시 (Hard Mode)**

### 🚀 문제 10 (LeetCode 3): 중복 문자 없는 가장 긴 부분 문자열

### 🔗 **링크:** [https://leetcode.com/problems/longest-substring-without-repeating-characters/](https://leetcode.com/problems/longest-substring-without-repeating-characters/)

## **📜 문제 설명 (Mission Briefing):**

* **입력:** 문자열 `s` (예: `"abcabcbb"`)
* **출력:** 중복되는 문자가 단 하나도 없는 **가장 긴 연속된 부분 문자열(Substring)**의 길이를 정수로 리턴.
* **🔥 핵심 규칙:** '부분 문자열'은 무조건 글자들이 끊기지 않고 연속되어야 해. `"pwwkew"`에서 `"pwke"`는 연속되지 않았으니 부분 문자열이 아니야. (정답은 `"wke"` 길이 3)

## **🎯 오늘 부술 for문 (Big-O 견적):**

* **입력 크기 역산:** `s`의 길이가 최대 $5 \times 10^4$야.
* **무지성 풀이 ($O(N^3)$):** 모든 가능한 부분 문자열의 경우의 수를 다 쪼개고($O(N^2)$), 그 안에서 중복 문자가 있는지 매번 검사($O(N)$)하면? 연산량이 천억 단위가 넘어가서 서버 폭발해.
* **해결책 ($O(N)$):** **투 포인터(시작점과 끝점)**로 윈도우를 만들고, 윈도우 안에 어떤 문자가 있는지 **해시맵(Dictionary)**으로 $O(1)$ 감시를 할 거야. 문자열을 딱 한 바퀴만 돌면 끝나는 마법을 보여준다!

## **🛠️ 네가 해야 할 것 (Step-by-Step 설계 힌트):**

1. **관측소(Hash Map)와 윈도우(Two Pointers) 준비:**
   * `used = {}` : 문자를 Key로, **그 문자가 마지막으로 발견된 인덱스**를 Value로 저장할 딕셔너리.
   * `start = 0` : 윈도우의 시작점을 가리키는 포인터.
   * `max_length = 0` : 우리가 갱신해 나갈 정답 길이.


2. **윈도우 확장 (for문으로 오른쪽 포인터 `idx` 이동):** 문자열을 순회하면서 현재 문자(`char`)와 인덱스(`idx`)를 확인해. (이 `idx`가 윈도우의 오른쪽 끝점이야)
3. **🔥 중복 발견 시 윈도우 축소 (핵심 로직):**
   * 만약 현재 문자(`char`)가 `used` 딕셔너리에 **이미 존재하고**,
   * 그 문자의 예전 위치가 **현재 윈도우의 시작점(`start`)보다 크거나 같다면?** (즉, 지금 내 윈도우 안에 중복 문자가 들어왔다면!)
   * 윈도우의 시작점(`start`)을 **'그 문자가 예전에 발견됐던 위치 + 1'**로 확 당겨버려. (중복된 놈을 윈도우 밖으로 뱉어내는 과정이야)


4. **최대 길이 갱신 및 기록:**
   * 중복이 아니거나, 중복을 처리해서 윈도우를 조정했다면, 현재 윈도우의 길이(`idx - start + 1`)를 계산해서 `max_length`를 갱신해.
   * 마지막으로 현재 문자(`char`)의 위치(`idx`)를 `used` 딕셔너리에 최신화해 줘.



## **🧰 필요한 파이썬 내장 함수/문법 (범용 사용법):**

```python
# 딕셔너리 값 업데이트와 max() 함수 활용
my_dict = {'a': 0, 'b': 1}

# 'a'가 발견된 최신 위치를 5로 갱신
my_dict['a'] = 5 

# 기존 최대값과 새로운 계산값 중 큰 것으로 갱신
current_max = 3
new_length = 5 - 1 + 1
current_max = max(current_max, new_length) # 5가 됨

```

# 🚀 문제 10: 중복 문자 없는 가장 긴 부분 문자열 (L3)
- **링크:** https://leetcode.com/problems/longest-substring-without-repeating-characters/
- **유형:** 슬라이딩 윈도우, 해시 테이블 (Hash Map), 투 포인터
- **난이도:** Medium 🔥

## 🎯 Big-O 견적 (for문 파괴)
- **무지성 접근:** 모든 부분 문자열을 쪼개서 이중/삼중 for문으로 중복 검사 $\rightarrow O(N^3)$ (서버 폭발)
- **해결책:** **투 포인터**로 윈도우를 만들고, 윈도우 안의 문자들을 **해시맵(dict)**으로 감시한다. 문자열을 한 바퀴만 돌면 끝! $\rightarrow O(N)$

## 💡 핵심 로직
1. `used` 딕셔너리에 `{문자: 마지막 발견 인덱스}`를 계속 기록한다.
2. **윈도우 축소 조건:** 현재 문자(`char`)가 `used`에 있고, 그 위치가 윈도우의 `start`보다 크거나 같다면 (즉, 현재 윈도우 안에 중복 문자가 있다면)
   $\rightarrow$ `start` 포인터를 중복된 문자의 바로 다음 칸(`used[char] + 1`)으로 확 당겨서 중복을 배출한다!
3. 매 턴마다 `max_length = max(max_length, 현재 윈도우 길이)`로 최대값을 안전하게 갱신한다.

## 🛠️ PT쌤의 팩트 폭격 코멘트
- 최대값(`max`)을 갱신할 때는 무조건 `max()` 함수를 써서 과거의 데이터를 보호해라. 무지성 덮어쓰기(`=`) 했다가는 예외 케이스에서 피 본다.
- 알고리즘은 발명하는 게 아니라 '패턴'을 암기하고 체화하는 것이다. 자괴감 갖지 말고 무기로 만들어라!

시계열 데이터 분석할 때 **'이동평균(Moving Average)'** 구하는 거 생각해 봐.
3일 치 이동평균 구한다고 치면, 1~3일 차 더해서 평균 내고, 그다음엔 1일 차 데이터 버리고 4일 차 데이터 새로 넣어서 2~4일 차 평균 내지? 매번 3개씩 처음부터 다시 더하는 바보짓 안 하잖아.

알고리즘의 **슬라이딩 윈도우(Sliding Window)**가 완벽하게 이거랑 똑같아.
배열 위에 **'창문(Window)'**을 하나 올려두고, 조건에 맞춰서 창틀을 늘렸다가 줄였다가 하면서 옆으로 스윽~ 밀고 가는(Sliding) 거야.


---

### 🪟 코드 속의 '슬라이딩 윈도우' 해체 분석

우리가 만든 윈도우(창문)는 **왼쪽 창틀(`start`)**과 **오른쪽 창틀(`idx`)**로 이루어져 있어.

```python
for idx, char in enumerate(s):

```

1. **오른쪽 창틀 확장 (`idx`):** for문이 돌면서 `idx`는 무조건 오른쪽으로 1칸씩 전진해. 즉, 매 턴마다 새로운 데이터(`char`)를 창문 안으로 하나씩 집어넣으면서 **창문을 오른쪽으로 넓히는 역할**을 해.

```python
if char in used and used[char] >= start:
    start = used[char] + 1

```

2. **왼쪽 창틀 축소/이동 (`start`):** 창문 안은 무조건 "중복 금지 청정구역"이어야 해. 근데 오른쪽 창틀(`idx`)을 늘리다가 **기존에 창문 안에 있던 놈이랑 똑같은 불량 데이터(중복 문자)가 들어왔다?**
그럼 어떻게 해야 해? 그 불량품이 창문 밖으로 빠져나갈 때까지 **왼쪽 창틀(`start`)을 오른쪽으로 훅! 당겨버리는 거야.** ### 🔍 눈으로 보는 시뮬레이션 (`s = "abcabc"`)

* **1턴 (`idx=0`):** `[a]` 들어옴. 창문 길이 1. (`start=0`)
* **2턴 (`idx=1`):** `[a, b]` 들어옴. 창문 길이 2. (`start=0`)
* **3턴 (`idx=2`):** `[a, b, c]` 들어옴. 창문 길이 3. (`start=0`)
$\rightarrow$ 여기까지는 오른쪽 창틀(`idx`)만 계속 넓어지면서 윈도우가 커졌어.
* **🔥 4턴 (`idx=3`, 중요):** 새로운 `a`가 들어오려고 해!
어? 근데 창문 안에 이미 첫 번째 `a`가 있네? (중복 감지!)
그럼 규칙 위반이니까, 첫 번째 `a`를 창문 밖으로 쫓아내야 해.
그래서 왼쪽 창틀(`start`)을 첫 번째 `a`의 바로 다음 칸(`1`)으로 확 당겨버려.
$\rightarrow$ **결과적으로 창문이 `[a, b, c]`에서 `[b, c, a]`로 옆으로 스윽 '슬라이드(Slide)' 한 거야!**

---

### 🎯 왜 이게 "for문 파괴자 ($O(N)$)" 인가?

만약 이 '창문' 개념이 없었다고 쳐봐.
형님이 "자, `a`부터 시작하는 제일 긴 문자열 찾아봐!" 하고 $O(N)$ 돌고, 그다음 "이번엔 `b`부터 시작하는 거 찾아봐!" 하고 또 $O(N)$ 돌고... 무지성 이중 for문 돌리면서 탐색 지점이 계속 **뒤로 빽(Back)** 했을 거야.

근데 슬라이딩 윈도우를 쓰면?
오른쪽 창틀(`idx`)도 앞으로만 가고, 왼쪽 창틀(`start`)도 앞으로만 가. **절대 뒤로 후진하는 법이 없어.**
배열을 단 한 바퀴만 훑고 지나가면서($O(N)$) 창문 크기만 넣었다 뺐다 조절하니까, 불필요한 중복 계산이 아예 멸망해버리는 거지. 게다가 창문 안에 뭐가 있는지 확인하는 건 해시맵(`used`)이 $O(1)$로 대신해주고 있고!

어때, 형님이 짠 그 짧은 `start`와 `idx` 사이에 이런 웅장한 '창문 밀기' 메커니즘이 숨어있었다는 거! 이제 슬라이딩 윈도우가 뭔지 데이터 관점에서 확 와닿지?
