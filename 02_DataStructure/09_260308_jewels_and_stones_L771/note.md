* **STEP 2 (해시/스택):** 이제부터는 **"메모리(공간)를 돈 주고 사서 시간(속도)을 찢어버리는"** 자본주의 싸움이야. 빈 딕셔너리(`{}`), 빈 리스트(`[]`), 빈 셋(`set()`) 같은 새로운 저장소를 팍팍 만들어. 데이터 크기가 커지면 메모리 좀 더 쓰는 건 알 빠 아니고, 무조건 탐색 속도를 **$O(1)$**로 만드는 게 지상 과제야. (이를 알고리즘에선 **Time-Space Trade-off**라고 해)

#### 3. 스택/큐(Stack/Queue): "시간의 순서를 기억하라" ⏳

# **2단계: 문제 및 가이드 제시 (Hard Mode)**

### 🚀 문제 771: 보석과 돌 (Jewels and Stones)

### 🔗 **링크:** [https://leetcode.com/problems/jewels-and-stones/](https://leetcode.com/problems/jewels-and-stones/)

## **📜 문제 설명 (Mission Briefing):**

* **입력:** `jewels` (보석의 종류를 나타내는 문자열), `stones` (내가 가진 돌멩이들을 나타내는 문자열)
* **출력:** 내가 가진 돌(`stones`) 중에 보석(`jewels`)이 도대체 몇 개나 섞여 있는지 카운트해서 정수로 리턴.
* **🔥 핵심 규칙:** 대소문자를 철저히 구분해! (예: `'a'`는 보석이지만 `'A'`는 똥돌일 수 있음)

## **🎯 오늘 부술 for문 (Big-O 견적):**

* **무지성 풀이 ($O(J \times S)$):** 돌멩이 하나 집어 들고 "이게 보석 목록에 있나?" 하고 `jewels`를 매번 처음부터 끝까지 뒤지면? $J$ 길이와 $S$ 길이를 곱한 만큼 탐색해야 해. 사실상 $O(N^2)$ 느낌이지.
* **해결책 ($O(J + S)$):** `jewels`를 **해시 테이블**에 때려 박아서 "탐색 시간"을 $O(1)$로 만들 거야! 그럼 보석 세팅할 때 $O(J)$, 돌멩이 검사할 때 $O(S)$로 끝나니까, 전체 시간 복잡도가 단일 for문 수준인 **$O(N)$**으로 확 떨어져.

## **🛠️ 네가 해야 할 것 (Step-by-Step 설계 힌트):**

1. **보석 도감(Hash) 만들기:** `jewels` 문자열을 탐색 속도가 $O(1)$인 파이썬 자료구조로 변환해. (저번처럼 딕셔너리를 써도 되지만, Key-Value 쌍이 필요 없고 그냥 **"존재 여부"**만 알면 될 땐 뭐가 제일 좋을까?)
2. **돌멩이 감정하기:** 내가 가진 `stones`를 for문으로 하나씩 돌면서, 방금 만든 '보석 도감'에 존재하는지 확인해.
3. **카운트:** 존재하면 정답 카운트를 +1 해준다.

## **🧰 필요한 파이썬 내장 함수/문법 (범용 사용법):**

* **`set()` (집합):** 파이썬의 `set`은 내부적으로 해시 테이블로 구현되어 있어. 중복을 제거해 줄 뿐만 아니라, 특정 원소가 안에 있는지 `in`으로 찾을 때 시간 복잡도가 **$O(1)$**이야.
```python
# 범용 예시
my_list = ['A', 'B', 'C', 'C']
my_set = set(my_list) # {'A', 'B', 'C'} - 중복 제거 및 해시화

if 'B' in my_set: # O(1) 초고속 탐색!
    print("B가 존재함")

```

# 🚀 문제 771: 보석과 돌 (Jewels and Stones)
- **링크:** https://leetcode.com/problems/jewels-and-stones/
- **유형:** 해시 테이블 (Hash Table), Set
- **난이도:** Easy

## 🎯 Big-O 견적 (for문 파괴)
- **무지성 접근:** 문자열(List)에서 `in` 탐색을 하면 매번 $O(J)$가 걸리므로 전체 $O(J \times S)$
- **해결책:** `jewels`를 **해시맵(Set)**으로 변환하여 탐색 시간을 $O(1)$로 만든다! 전체 복잡도는 $O(J + S)$. 메모리(공간)를 써서 시간(속도)을 사는 전형적인 Trade-off 전략.

## 💡 핵심 로직
1. 파이썬의 `set()`은 내부적으로 해시 테이블로 구현되어 있어, `in` 연산 시 압도적인 속도($O(1)$)를 자랑한다.
2. 어떤 원소가 '존재하는지' 여부만 판단할 때는 `dict`보다 `set`이 훨씬 가볍고 빠르다.

## 🐍 Pythonic Code
```python
class Solution:
    def numJewelsInStones(self, jewels: str, stones: str) -> int:
        # 1. 보석 도감을 Set(해시)으로 만들어 O(1) 탐색 준비
        jewels_set = set(jewels)
        
        # 2. 파이썬의 True=1, False=0 성질과 sum()을 이용해 한 줄 컷
        return sum(s in jewels_set for s in stones)
```









### 🔥 [심화] List Comprehension vs Generator Expression

* `sum([s in jewels_set for s in stones])`: 대괄호 `[]`를 쓰면 메모리에 리스트를 몽땅 다 만들어둔 뒤에 합계를 구함. (데이터가 커지면 메모리 낭비 심함, 공간 복잡도 $O(S)$)
* `sum(s in jewels_set for s in stones)`: 대괄호를 빼면 **제너레이터 표현식(Generator Expression)**이 되어, 하나씩 꺼내서 평가하고 버리는 방식(On-the-fly)으로 동작함. (메모리 거의 안 씀, 공간 복잡도 $O(1)$)
* **결론:** 집계 함수(`sum`, `max`, `min` 등) 안에서는 가급적 대괄호를 빼고 제너레이터를 써라!

## 🛠️ PT쌤의 팩트 폭격 코멘트

* 앞으로 "이 값이 저 목록에 있나?" 찾을 땐 무지성 리스트 탐색하지 말고, **조건반사처럼 `set()`을 씌워서 해시 탐색**으로 찢어버리자.
